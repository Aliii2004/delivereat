P1 - The Relationship Between an API and a Software Development Kit (SDK)
================================================================================

An Application Programming Interface (API) is a defined contract between two software
components that specifies how they communicate. An API exposes a set of endpoints,
methods, or functions that a consuming application may call, without requiring knowledge
of the internal implementation of the service providing those functions. The API defines
what operations are available, what data must be sent, and what will be returned.

A Software Development Kit (SDK) is a collection of pre-built tools, libraries,
documentation, and code samples designed to simplify the use of an API within a specific
programming language or environment. Where an API defines the rules of communication,
an SDK provides a ready-made implementation of those rules.

The distinction is most clearly understood through a practical example from the DeliverEat
platform. The Socket.io server-side API defines events such as "join:order", "order:status",
and "courier:moved", along with the data structures each event carries. This is the API
contract. The socket.io-client npm package, imported in client/src/lib/socket.ts, is the SDK.
It provides the io() constructor, automatic reconnection logic, transport negotiation
(WebSocket with polling fallback), and event listener management. A developer using only
the raw API contract would need to implement all of this manually.

From client/src/lib/socket.ts:

    const { io } = require('socket.io-client');
    socket = io(SOCKET_URL, {
      auth: { token },
      transports: ['websocket', 'polling'],
      reconnection: true,
      reconnectionAttempts: 5,
      reconnectionDelay: 1000,
    });

Without the SDK, establishing this connection would require manually constructing the
WebSocket upgrade request, handling transport fallback, managing the authentication
handshake, and implementing exponential backoff for reconnection. The SDK abstracts
all of this into a single function call.

A further example is the Prisma ORM, which serves as an SDK for the PostgreSQL API.
The raw PostgreSQL API accepts SQL strings over a TCP connection. Prisma wraps this
into a type-safe TypeScript client where developers call prisma.order.create() with a
typed object, and the SDK generates the corresponding parameterised SQL query. This
eliminates an entire class of SQL injection vulnerabilities and provides compile-time
type checking on all database operations.

The relationship between API and SDK is therefore hierarchical and complementary. An API
defines the specification; an SDK implements that specification in a developer-friendly form
for a particular language or runtime. The API is language-agnostic; the SDK is
language-specific. An application can interact with an API without an SDK, but doing so
requires significantly more implementation effort and introduces a higher risk of
protocol-level errors.

M1 - Assessment of a Range of APIs for the DeliverEat Platform
================================================================================

DeliverEat uses four distinct API types, each selected to address specific technical
requirements. Each is assessed below in terms of design, capability, and suitability.


REST API (Server 1 - Express/Node.js)

REST (Representational State Transfer) is an architectural style for distributed
hypermedia systems. REST APIs use standard HTTP verbs (GET, POST, PATCH, DELETE),
stateless communication, and resource-based URLs. Server 1 implements over 30 REST
endpoints across authentication, orders, menus, couriers, reviews, and notifications.

The authentication routes in server1/src/routes/auth.routes.ts demonstrate the REST
implementation pattern:

    router.post('/register',
      [
        body('email').isEmail().normalizeEmail(),
        body('password').isLength({ min: 6 }),
        body('role').isIn(['CUSTOMER', 'RESTAURANT_OWNER', 'COURIER']),
      ],
      validate,
      authController.register
    );

Each route applies input validation at the routing layer before the request reaches
the controller. This ensures that malformed data is rejected with descriptive error
messages before any database operations are attempted.

Strengths: Universally understood, cacheable responses via HTTP headers, straightforward
to test using tools such as Postman or curl, aligns naturally with CRUD operations.

Limitations: Clients may receive more data than required (over-fetching), related
resources require multiple round-trips, and there is no native mechanism for server-push
or real-time updates.


GraphQL API (Server 2 - FastAPI/Python with Strawberry)

GraphQL is a query language and server-side runtime for APIs developed by Meta.
Unlike REST, the client specifies exactly which fields it requires in each request.
Server 2 exposes analytics data through a single /graphql endpoint.

From server2/app/graphql/schema.py:

    @strawberry.type
    class Query:
        @strawberry.field(description="Restaurant statistics")
        async def restaurant_stats(
            self, restaurant_id: str, days: Optional[int] = 30
        ) -> RestaurantStatsType:
            async with AsyncSessionLocal() as db:
                service = StatsService(db)
                return RestaurantStatsType(**await service.get_restaurant_stats(
                    restaurant_id, days or 30
                ))

The schema is defined using Python decorators. Each resolver opens its own database
session using async context management, ensuring sessions are closed correctly after
each query regardless of whether the query succeeds or raises an exception.

Strengths: Eliminates over-fetching, strongly typed schema with automatic documentation,
single endpoint regardless of query complexity, introspection allows clients to discover
available operations.

Limitations: Query depth attacks can cause exponential database load if depth limiting
is not enforced, caching is more complex than REST, and the learning curve is steeper
for developers familiar only with REST conventions.


WebSocket API - Socket.io

Socket.io provides bidirectional, event-based real-time communication over WebSocket
with an HTTP long-polling fallback. DeliverEat uses Socket.io for live order status
updates, courier GPS broadcasting, surge pricing alerts, and push notifications.

The client connects with JWT authentication on the handshake:

    socket = io(SOCKET_URL, {
      auth: { token },
      transports: ['websocket', 'polling'],
      reconnection: true,
      reconnectionAttempts: 5,
    });

The server uses room-based messaging so that events are only delivered to clients
with a legitimate interest. When a courier updates their GPS position, the event
is emitted only to the specific order room, not broadcast to all connected clients.
When a new order is placed, the notification is sent only to the restaurant room for
that particular restaurant.

Strengths: Full-duplex communication, low latency, targeted delivery via rooms,
persistent connection reduces per-message overhead compared to repeated HTTP polling.

Limitations: Stateful connections require Redis adapter for horizontal scaling,
not suitable for cacheable or idempotent operations, WebSocket connections consume
server memory for each connected client.


Internal HTTP API - Microservice Communication

Server 2 exposes a POST /internal/order-completed endpoint consumed by Server 1
after a delivery is confirmed. This triggers analytics recording in the separate
analytics database without Server 1 needing direct access to it.

Authentication uses a shared secret passed as an X-Internal-Secret header:

    INTERNAL_API_SECRET=BotJIQKb4a7oyIUkSa7SB7SQvzHQFyvCTxIm3VVN4K8=

Alongside this synchronous HTTP call, Server 1 also publishes to the Redis
order.events channel. Server 2's redis_subscriber consumes these events
asynchronously. This dual-mechanism design means analytics data is captured
even if one transport experiences a transient failure.

Strengths: Clear service boundary, failure in analytics service does not block
order fulfilment, shared secret provides lightweight authentication suitable for
internal network communication.

Limitations: Synchronous HTTP calls introduce latency into the delivery confirmation
flow if Server 2 is slow to respond. The asynchronous Redis path mitigates this by
providing an eventual-consistency fallback.

D1 - Evaluation of Security Issues Surrounding APIs
================================================================================

The OWASP API Security Top 10 provides a recognised framework for evaluating API
security risks. The following sections assess each major vulnerability category against
the DeliverEat implementation.


Broken Object Level Authorisation (OWASP API1)

This vulnerability occurs when an API accepts a user-supplied resource identifier but
does not verify that the requesting user is authorised to access that resource. It is
ranked first in the OWASP API Security Top 10 because it is the most commonly exploited
API vulnerability in practice.

In DeliverEat, every order access endpoint performs explicit ownership verification:

    const isCustomer        = order.customerId === userId;
    const isCourier         = role === 'COURIER' && order.courier !== null;
    const isRestaurantOwner = role === 'RESTAURANT_OWNER' &&
                              order.restaurant.ownerId === userId;

    if (!isCustomer && !isCourier && !isRestaurantOwner && !isAdmin) {
        throw new AppError("Access denied", 403);
    }

Without this check, any authenticated user could access any order by supplying a
valid UUID. The UUID format (128-bit random identifier) does not provide security by
obscurity - with sufficient requests, an attacker could enumerate valid order IDs.
The explicit ownership check ensures that even correctly formatted UUIDs are rejected
if the requesting user does not own the resource.


Broken Authentication (OWASP API2)

DeliverEat uses JSON Web Tokens for both REST and WebSocket authentication. Several
attack vectors apply to JWT-based systems.

Token theft and replay: If an access token is intercepted, an attacker can use it
until expiry. Mitigation: access tokens expire after 15 minutes. Refresh tokens are
hashed with SHA-256 before storage in Redis - the plaintext token is never persisted.
On logout, the token hash is added to a blacklist; subsequent requests using the
same token are rejected regardless of expiry status.

Weak secrets: If JWT_SECRET is short or predictable, offline brute-force attacks
against intercepted tokens become feasible. The application validates secret strength
at startup:

    if ((process.env.JWT_SECRET?.length ?? 0) < 32) {
        console.error('JWT_SECRET must be at least 32 characters');
        process.exit(1);
    }

The actual secret in production is a 64-character base64-encoded random value,
providing approximately 384 bits of entropy.

Algorithm confusion: Setting the algorithm field in a JWT header to "none" can
bypass signature verification in vulnerable libraries. The jsonwebtoken library
used in this project does not accept unsigned tokens by default, and the algorithm
is explicitly specified in all verification calls.


Price Tampering (OWASP API3 - Broken Object Property Level Authorisation)

In e-commerce applications, accepting item prices from the client is a common
vulnerability. An attacker could modify the request body to set prices to zero or
negative values before submitting an order.

DeliverEat addresses this by ignoring any price values in the order creation request.
The server fetches authoritative prices from the database:

    const menuItems = await prisma.menuItem.findMany({
        where: {
            id: { in: menuItemIds },
            restaurantId,
            isAvailable: true,
        },
    });
    const linePrice = menuItem.price * item.quantity;

The restaurantId constraint in the WHERE clause also prevents a cross-restaurant
attack, where an attacker submits items from a cheap restaurant through a more
expensive restaurant's checkout endpoint.


Race Condition in Order Pickup (OWASP API4 - Unrestricted Resource Consumption)

Without concurrency protection, two couriers submitting simultaneous pickup requests
for the same order could both receive a success response, creating a data integrity
problem where a single order is assigned to multiple couriers.

The implementation uses an atomic database operation to prevent this:

    const result = await prisma.order.updateMany({
        where: {
            id: orderId,
            status: OrderStatus.READY,
            courierId: null,
        },
        data: {
            status: OrderStatus.ON_THE_WAY,
            courierId: courier.id,
        },
    });

    if (result.count === 0) {
        throw new AppError('Order is no longer available', 409);
    }

The WHERE clause checks status: READY and courierId: null atomically. Once the first
courier's request sets courierId to a value, all subsequent concurrent requests find
courierId is no longer null and receive a 409 Conflict response. This pattern is
equivalent to an optimistic lock and is the correct approach for this category of
concurrency problem.


GraphQL-Specific Vulnerabilities (OWASP API4)

GraphQL introduces two security concerns not present in REST APIs.

Introspection exposure: GraphQL's introspection feature allows any client to query
the full schema, revealing all available types, fields, and relationships. In production
this provides attackers with a detailed map of the data model. Mitigation in main.py:

    graphql_router = GraphQLRouter(
        schema,
        graphiql=os.environ.get("NODE_ENV") != "production",
    )

The interactive explorer is disabled in production, but introspection via direct POST
requests is not yet disabled. This should be addressed using a Strawberry extension
that blocks introspection queries when NODE_ENV equals "production".

Query depth attacks: A deeply nested GraphQL query can trigger exponential database
operations. For example, querying events within a restaurant that include order details
which include courier details could create O(n^3) database queries. DeliverEat does
not currently implement query depth limiting, which represents an outstanding
vulnerability. The recommended mitigation is to restrict query depth to a maximum
of 7 levels using a custom Strawberry extension.


Internal API Secret Management (OWASP API8 - Security Misconfiguration)

The INTERNAL_API_SECRET is stored as an environment variable, correctly excluded
from source control via .gitignore. The .env.example file documents the variable
name without a value, which is the accepted industry practice for communicating
required configuration without exposing secrets.

A risk remains if the secret is committed accidentally. The project would benefit
from a pre-commit hook using a tool such as detect-secrets to scan staged files for
high-entropy strings before allowing commits.


Summary of Security Posture

The DeliverEat API implementation addresses the majority of OWASP API Security Top 10
categories with documented, testable mitigations in the source code. The two outstanding
vulnerabilities - GraphQL query depth limiting and rate limiting on REST endpoints -
represent medium-severity risks in a production environment. Both have clear and
well-understood remediation paths that should be implemented prior to any public
deployment.

================================================================================

DeliverEat was built upon an initial application skeleton comprising Docker Compose
service definitions, an Express server with basic routing, a Next.js frontend with
placeholder pages, and PostgreSQL and Redis containers. No business logic or API
integration existed in the original framework.

The following APIs were integrated into this existing structure.

REST API on Server 1: The Express skeleton was extended with six route groups
registered in server1/src/index.ts:

    app.use('/api/auth',          authRoutes);
    app.use('/api/orders',        orderRoutes);
    app.use('/api/menu',          menuRoutes);
    app.use('/api/couriers',      courierRoutes);
    app.use('/api/reviews',       reviewRoutes);
    app.use('/api/notifications', notificationRoutes);
    app.use('/api/addresses',     addressRoutes);

Each route group follows the same middleware chain: input validation using
express-validator, authentication middleware enforcing JWT verification, and
role-based access control before reaching the controller.

GraphQL API on Server 2: The FastAPI service was extended with a Strawberry GraphQL
schema defining four resolvers. The internal webhook endpoint at
POST /internal/order-completed was added to receive delivery completion events from
Server 1. The service is initialised with a lifespan context manager that starts the
Redis subscriber on startup and disconnects it cleanly on shutdown.

Socket.io integration: The Socket.io server was initialised within the Express
application by passing the HTTP server instance to the initSocket function. JWT
authentication middleware was applied to all socket connections. Room management
ensures that events are only delivered to clients with authorisation to receive them.

Redis extension: Redis was used both as a key-value cache (GPS locations, surge
multipliers, token blacklists, order status) and as a Pub/Sub event bus. The
order.events channel allows Server 1 to notify Server 2 of lifecycle events without
a direct dependency between the two services.

M2 / D2 - API Application Design with Justification
================================================================================

The choice to use four distinct API types reflects deliberate architectural decisions
rather than technical convenience.

REST was selected for CRUD operations because it is stateless, cacheable, and aligns
naturally with the request-response pattern that browsers and HTTP clients expect.
For operations such as creating a menu item, retrieving a restaurant list, or updating
a courier's status, there is no requirement for real-time communication or flexible
field selection. The simplicity of REST reduces client implementation complexity and
makes the API straightforward to document and test.

An alternative considered was using GraphQL for all operations, including CRUD. This
was rejected because GraphQL introduces unnecessary complexity for simple create and
update operations, and because REST's HTTP caching behaviour is valuable for public
endpoints such as GET /api/menu/restaurants, which multiple clients will call with
identical parameters.

GraphQL was selected exclusively for analytics because analytics queries vary
significantly by user role and dashboard context. A restaurant owner's dashboard
requires total revenue, order volume, and a time-series burndown. A courier's
dashboard requires delivery count, average delivery time, and earnings. Under REST,
this would require either separate endpoints for each dashboard type, or a single
endpoint that returns all possible fields regardless of what the client needs.
GraphQL resolves this by allowing each client to specify exactly the fields required
in the query. The separation of analytics onto a dedicated Python service also means
that computationally intensive aggregation queries run on Server 2 without affecting
the Node.js event loop on Server 1.

Socket.io was selected over Server-Sent Events (SSE) because DeliverEat requires
bidirectional communication. Couriers must be able to both receive order assignments
and upload GPS positions. SSE supports only server-to-client streaming. Socket.io's
room model also provides a natural mapping to the application's domain: each active
order has a room, each restaurant has a room, and a shared "all-zones" room serves
couriers receiving surge alerts.

An alternative to Socket.io considered was direct WebSocket without the Socket.io
abstraction layer. This was rejected because Socket.io's built-in features - transport
negotiation, automatic reconnection, room management, and event namespacing - would
all need to be reimplemented manually. The added abstraction cost in terms of bundle
size is justified by the reduction in implementation risk.

Redis Pub/Sub was chosen over direct HTTP from Server 1 to Server 2 for asynchronous
events because it introduces loose coupling. If Server 2 is temporarily unavailable,
published messages queue in Redis and are consumed when Server 2 recovers. Direct HTTP
would cause Server 1 to block waiting for Server 2 to respond, introducing latency
into the order fulfilment path. The synchronous internal HTTP endpoint at
/internal/order-completed is retained as a secondary confirmation mechanism for
time-sensitive analytics updates.

P3 / M3 / D3 - Implementation Using Multiple APIs
================================================================================

REST API Implementation

All REST routes follow a consistent middleware chain:

    Request → Nginx → Express Router → Validation Middleware → Auth Middleware → Controller → Prisma → PostgreSQL → Response

Input validation is performed using express-validator's body() chain. The validate
middleware calls validationResult() and returns all validation errors in a single
response, rather than stopping at the first error. This provides better feedback to
API consumers.

The order creation endpoint demonstrates how multiple APIs are orchestrated within
a single REST handler:

    // 1. Database read - authoritative price lookup
    const menuItems = await prisma.menuItem.findMany({
        where: { id: { in: menuItemIds }, restaurantId, isAvailable: true }
    });

    // 2. Redis read - surge pricing cache
    const surgeMultiplier = await SurgeService.getSurgeMultiplier(restaurantId);
    const deliveryFee = Math.round(15000 * surgeMultiplier);

    // 3. Database write - order persistence
    const order = await prisma.order.create({ data: { ...orderData, deliveryFee } });

    // 4. Redis Pub/Sub - async analytics notification
    await redisService.publish('order.events', { type: 'order_created', orderId: order.id });

    // 5. Socket.io - real-time restaurant notification
    getIO().to(`restaurant:${restaurantId}`).emit('order:new', { orderId: order.id });

    // 6. Database write - notification persistence
    await prisma.notification.create({ data: { userId: restaurant.ownerId, type: 'NEW_ORDER' } });

A single REST call to POST /api/orders coordinates six separate API interactions
across three technologies. This demonstrates API composition as an architectural
pattern.


GraphQL Implementation

The analytics schema uses async session management for each resolver, preventing
connection pool exhaustion under concurrent load. Each resolver is responsible for
opening and closing its own database session:

    @strawberry.field(description="Courier performance report")
    async def courier_performance(
        self, courier_id: str, days: Optional[int] = 30
    ) -> CourierPerformanceType:
        async with AsyncSessionLocal() as db:
            service = StatsService(db)
            data = await service.get_courier_performance(courier_id, days or 30)
            await db.commit()
            return CourierPerformanceType(**data)

The async with block guarantees that the session is committed on success and rolled
back on exception, regardless of which code path is taken.


Socket.io Implementation

The Socket.io server validates JWT tokens on every connection using middleware applied
before any event handlers are registered:

    io.use((socket, next) => {
        const token = socket.handshake.auth.token;
        const decoded = jwt.verify(token, process.env.JWT_SECRET);
        socket.data.userId = decoded.userId;
        socket.data.role = decoded.role;
        next();
    });

When a client emits join:order, the server verifies that the requesting user owns the
order before adding the socket to the order room. This prevents a malicious client
from subscribing to events for orders they do not own. Similarly, join:restaurant
verifies that the user is the owner of that restaurant, and join:all-zones verifies
that the user has the COURIER role.

The client reconnection logic in socket.ts updates the auth token on reconnect,
ensuring that a reconnection following an access token refresh uses the new token
rather than the expired one:

    socket.on('disconnect', (reason) => {
        if (reason !== 'io server disconnect' && reason !== 'io client disconnect') {
            const newToken = Cookies.get('accessToken');
            if (socket && newToken) {
                socket.auth = { token: newToken };
            }
        }
    });

M5 / D4 - Application Updates, Critical Evaluation, and Security Report
================================================================================

Updates Applied Based on Testing Results

Three defects were identified during testing and subsequently resolved.

Defect 1: Courier ID mismatch causing empty order list for couriers
The getMyOrders controller filtered courier orders using WHERE courierId = userId,
where userId is the User table primary key. However, the Order.courierId field stores
the Courier table primary key, which is a different UUID. The fix adds a lookup of
the Courier record before filtering:

    const courier = await prisma.courier.findUnique({ where: { userId }, select: { id: true } });
    where = courier ? { courierId: courier.id } : { id: 'none' };

Defect 2: Available orders disappearing after GPS acquisition
When the browser provided a real GPS position, a React state update triggered a
useEffect re-fetch with the new coordinates. If the courier's physical location was
far from the restaurant (or a VPN was in use), the geo-filter returned zero results.
The fix replaced useState with useRef for the location value, preventing re-renders
on GPS update, and increased the search radius from 10km to 50km for the demonstration
environment.

Defect 3: Incorrect address endpoint in checkout
The checkout page fetched addresses via GET /auth/me, which does not consistently
include the addresses relation in the response. The fix replaced this with a direct
call to GET /api/addresses, which returns the correct address array.


Critical Evaluation of APIs

REST API

The REST implementation is functionally complete and follows established conventions
for HTTP status codes, input validation, and error response formatting. The middleware
architecture enforces a clear separation between validation, authentication, and
business logic, making each layer independently testable.

The primary weakness is the absence of rate limiting. Without rate limiting, the
endpoints are susceptible to brute-force attacks (particularly against POST /auth/login)
and denial-of-service through high request volume. Production deployment requires rate
limiting at the Nginx layer or via a middleware such as express-rate-limit, with stricter
limits applied to authentication endpoints.

API versioning is also absent. The current API path structure (/api/orders) contains
no version identifier. Any breaking change to the API contract would require all
consumers to update simultaneously. Introducing /api/v1/ prefixes would allow the
current API to remain available while a revised version is developed in parallel.

The authentication mechanism is well-implemented. The combination of short-lived access
tokens, hashed refresh token storage, and a Redis blacklist provides defence in depth
against the most common token-based attack vectors.


GraphQL API

The GraphQL implementation successfully separates analytics concerns from operational
data and demonstrates appropriate use of async session management to prevent connection
pool exhaustion.

Query depth limiting is the most significant outstanding security gap. Without it, a
client can submit a query with arbitrarily deep nesting that translates into an
exponential number of database operations. This should be addressed using a Strawberry
extension that rejects queries exceeding a defined depth threshold before any resolvers
are executed.

The conditional disabling of GraphiQL in production is correctly implemented, but
programmatic introspection via direct POST requests remains available. Full introspection
suppression in production would require an additional Strawberry extension.

The architecture decision to isolate analytics on a separate Python service is justified.
The async SQLAlchemy engine handles concurrent analytical queries more efficiently than
the Node.js event loop would, and the separation means that a slow or failing analytics
query has no impact on the operational API.


Socket.io API

The Socket.io implementation handles authentication, room management, and GPS event
broadcasting correctly. The reconnection configuration (5 attempts, 1 second initial
delay) provides reasonable resilience to transient network failures without creating
excessive reconnection storms under sustained outage.

The token refresh integration is well-designed. When the Axios interceptor refreshes
the access token, it calls reconnectSocketWithNewToken(), which disconnects the existing
socket, removes all listeners, sets the new token on the auth object, and establishes a
fresh authenticated connection. This ensures the WebSocket session uses the same
credentials as the HTTP session.

The stateful nature of WebSocket connections creates a scaling constraint. The current
single-instance deployment does not require a Redis adapter, but any horizontal scaling
would require socket.io/redis-adapter to synchronise room membership across instances.
This is a known limitation with a well-documented remediation path.
